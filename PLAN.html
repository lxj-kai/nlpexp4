RGB数据使用方案

RGB数据集的结构天然适合本实验——positive = 有用文档，negative = 噪音文档（语义相关但不含答案）。

数据构造方式

def build_experiment_sample(record, noise_ratio=0.5, noise_type="semantic"): """ record: RGB数据集的一条记录 noise_ratio: 噪音文档占比 (0.0 ~ 1.0) noise_type: "semantic" | "counterfactual" | "mixed" """ positives = record["positive"] # 包含正确答案的文档 negatives = record["negative"] # 语义相关但不含答案的文档 # 按比例采样噪音文档 n_pos = len(positives) n_neg = int(n_pos * noise_ratio / (1 - noise_ratio)) sampled_neg = random.sample(negatives, min(n_neg, len(negatives))) # 混合并打乱顺序 context_docs = positives + sampled_neg random.shuffle(context_docs) return { "query": record["query"], "context": context_docs, "answer": record["answer"], "noise_ids": list(range(n_pos, n_pos + len(sampled_neg))), "noise_ratio": len(sampled_neg) / len(context_docs) }

技术方案详细设计

噪音注入器设计

核心模块，控制实验中的噪音变量。

噪音维度

维度变量取值
噪音比例noise_ratio0%, 25%, 50%, 75%, 100%
噪音类型noise_type 语义噪音 SeN 语义相关不含答案 (RGB negative)
反事实噪音 CN 包含错误信息 (RGB positive_wrong)
混合噪音 Mixed 两者混合
噪音位置noise_position 前置 噪音放最前
后置 噪音放最后
穿插 随机分布（默认）
位置维度的灵感来自 "Lost in the Middle"：LLM对不同位置的信息关注度不同。噪音文档放在不同位置，影响可能截然不同。这是一个加分的实验设计。

评估指标体系（题目要求"提出评估指标"）

层次1：基础问答指标

指标计算方式衡量什么
Exact Match (EM)答案是否精确匹配严格准确率
Token-F1token级别precision/recall部分匹配质量
ROUGE-L最长公共子序列内容覆盖度
LLM-as-JudgeDeepseek打分(1-5分)语义准确性

层次2：噪音鲁棒性专用指标（自主提出）

指标名公式含义
噪音敏感度 (NS) NS = (Score_clean - Score_noisy) / Score_clean 性能相对下降幅度。NS越大 = 越容易被噪音干扰
噪音抵抗曲线斜率 (NRS) NRS = ΔScore / ΔNoiseRatio 噪音比例每增加1%，性能下降多少。斜率越平缓 = 越鲁棒
信息溯源率 (ISR) ISR = 可追溯到positive文档的关键信息数 / 答案中的总关键信息数 答案是否基于可靠文档生成。ISR低 = 模型在编造或被噪音误导
噪音采纳率 (NAR) NAR = 答案中来自negative文档的信息数 / 答案中的总信息数 模型在多大程度上"采信"了噪音文档
矫正恢复率 (CRR) CRR = (Score_corrected - Score_noisy) / (Score_clean - Score_noisy) 矫正机制恢复了多少损失。CRR=1 = 完全恢复；CRR>1 = 比无噪音更好
这5个自主提出的指标构成完整的噪音鲁棒性评估框架：NS和NRS衡量"影响有多大"，ISR和NAR衡量"影响机制是什么"，CRR衡量"矫正效果如何"。这是本项目的理论贡献。

矫正方法设计（3种 + 1个对比baseline）

方法A

提示词优化 (Prompt Correction)

原理：在prompt中显式引导LLM质疑检索文档

  • 要求模型判断每条文档是否与问题逻辑相关
  • 要求对关键结论标注来源文档编号
  • 遇到矛盾信息时要求指出

API调用：1次（无额外开销）

参考：类似CRAG的检索质量意识

方法B

迭代式过滤生成 (Iterative Filter)

原理：先评估文档质量→过滤噪音→再生成

  • Step1: LLM对每个文档评分(高/中/低相关性)
  • Step2: 过滤低相关文档
  • Step3: 基于过滤后文档生成答案
  • Step4(可选): 自我验证

API调用：2-3次

参考：CRAG的分解-重组思路

方法C

证据链推理 (CoT Evidence)

原理：要求LLM先构建证据链再回答

  • 强制LLM逐步推理：
  • 1.识别问题的关键信息需求
  • 2.逐个检查文档是否满足需求
  • 3.基于满足需求的文档构建答案

API调用：1次（prompt更长）

参考：CoT-RAG的链式推理

对比方法

Self-RAG思路复现

原理：模拟Self-RAG的反思机制

  • 让LLM判断"这个检索结果是否有用"
  • 让LLM判断"我的回答是否有文档支撑"
  • 如果不支撑，重新选择文档

API调用：2-4次

参考：Self-RAG的反思token机制

子实验：位置影响（在50%噪音比例下）

位置策略说明假设
前置噪音文档全部放在context开头由于primacy bias，影响较大
后置噪音文档全部放在context末尾由于recency bias，影响较大

实验三：案例深度分析

目标：精选15-20个典型案例，追踪噪音误导的全过程

每个案例需要展示：

项目内容
1. 问题原始query
2. 参考答案ground truth
3. 正确文档positive文档内容（标绿）
4. 噪音文档negative文档内容（标红）+ 标注为什么是噪音
5. 无噪音回答只给positive时的回答 ✓
6. 有噪音回答混入negative后的回答 ✗ + 标注错误来源
7. 矫正后回答矫正机制介入后的回答 + 标注矫正如何生效
8. 误导机制分析模型为什么被噪音误导？是复制了噪音信息？还是受干扰后自行编造？

案例分类

  • Type-1: 直接采信型 — 模型直接复制了噪音文档中的错误信息
  • Type-2: 干扰编造型 — 噪音导致模型"困惑"，自行编造了答案
  • Type-3: 信息淹没型 — 正确信息被大量噪音稀释，模型未找到正确答案
  • Type-4: 免疫型 — 即使有噪音，模型仍然回答正确（分析为什么）
